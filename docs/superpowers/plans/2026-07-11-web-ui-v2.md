# Cairn Web UI v2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Tasks use checkbox (`- [ ]`) syntax for tracking.
>
> **Plan style (per user direction):** Tasks are defined by *outcome, interfaces, and acceptance criteria* — not step-by-step edits. Implementers are expected to read the referenced spec/code, exercise judgment on internals, and handle unexpected scenarios sensibly (e.g., an API that moved in a dependency upgrade) rather than following rote instructions. When a task's constraints genuinely can't be met as written, stop and report rather than improvise around the spec.
>
> **Model guidance:** dispatch implementation subagents on the model named per task (sonnet minimum; opus for tasks marked so). Do not use haiku-class models for any task in this plan.

**Goal:** A browser UI (session list/create/detail + ghostty-compatible terminal attach) served by the daemon, speaking wRPC over WebSocket (primary) and WebTransport (secondary) via the official `@bytecodealliance/wrpc` JS SDK.

**Architecture:** The daemon gains a `ws://` HTTP/WebSocket listener (axum) feeding the existing `wrpc_transport::frame::Server` accept loop — one WS connection per invocation — plus orthogonal `--web-ui` SPA serving and a `/cairn.json` endpoint-discovery document. The web client is a SvelteKit SPA with a framework-free `protocol/` layer over the official JS SDK.

**Tech Stack:** Rust (axum, wrpc-transport 0.29, wrpc-websockets git dep), Node + Vite + SvelteKit + Svelte 5 + TypeScript + vitest, `@wterm/dom` + `@wterm/ghostty`, `@bytecodealliance/wrpc` (vendored).

**Spec:** `docs/superpowers/specs/2026-07-11-web-ui-v2-design.md` — every implementer should read it first; it is the authority on behavior. The v1 spec (`2026-06-13-web-ui-design.md`) and the `feat/web-ui` branch (worktree at `.worktrees/failed-web-ui`) are *reference only, not authoritative* — v1's hand-written codec approach is abandoned.

## Global Constraints

- No `panic!`/`unwrap()`/`expect()` in production Rust; return errors (test code exempt). Applies to TS analogues: no swallowed promise rejections.
- Rust tests run with `cargo nextest run`; lint gate is `cargo clippy --all-targets -- -D warnings` + `cargo fmt --check`.
- Never log PTY byte contents (established observability rule).
- Locking discipline (daemon): never hold an entry lock across `.await`.
- Pre-commit runs `biome` and a `svelte-build` hook on `cairn-web/src/**` files, and clippy/fmt on Rust — commits must pass them.
- Commit at each task boundary at minimum; messages explain *why* per repo convention.
- The wrpc git pin (Rust `wrpc-websockets` dep) and the vendored JS SDK must reference the same wrpc commit. Local checkout for reference: `../wrpc`.
- JS SDK wire model: **one duplex byte stream per invocation**; binary WS frames carry data, an empty *text* frame is the EOF sentinel.

### Task 1: wrpc dependency upgrade — **sonnet**

**Files:** `Cargo.toml` (workspace), `crates/*/Cargo.toml`, whatever the API churn touches (expect `serve/wrpc.rs`, `serve/transport/*.rs`, `cairn-protocol/src/lib.rs`, `cairn-client/src/connect.rs`).

**Outcome:** Workspace builds and all existing tests pass on `wrpc-transport 0.29`, `wrpc-transport-web 0.3`, and the matching `wit-bindgen-wrpc` release. Consult `../wrpc` git history/CHANGELOG for breaking changes rather than guessing at compile errors.

**Acceptance:**
- [ ] Pins bumped; `cargo tree` shows no duplicate wrpc-transport versions
- [ ] `cargo nextest run` fully green; clippy/fmt clean
- [ ] No behavioral workarounds — if 0.29 changed semantics (not just names), surface it in the task report

### Task 2: `ws://` transport listener — **opus** (blocked by Task 1)

**Files:** create `serve/transport/websocket.rs`, `serve/http.rs`; modify `listen.rs`, `serve/transport/mod.rs`, `config/{args,mod}.rs`, daemon `Cargo.toml` (axum; `wrpc-websockets` as git dep pinned to a commit — record the pin choice); test `crates/cairn-daemon/tests/` (new WS integration tests using `wrpc_websockets::Client` as the test-side dialer).

**Interfaces produced:** `ListenerConfig::WebSocket { addr }` parsed from `ws://host:port` (`localhost` expands dual-stack like WT does, `listen.rs:55`); an axum server exposing `/ws` (upgrade → one wRPC invocation via `wrpc_websockets::split` → existing `frame::Server` accept path in `serve/wrpc.rs`) and `/healthz`; `wss://`/`https://`/`unix` behavior unchanged.

**Origin validation is part of this task** (spec: "Identity & auth"): upgrades carrying an `Origin` header must match the request's own `Host`-derived origins or a repeatable `--ws-origin <origin>` allowlist; absent `Origin` passes; mismatches are rejected before upgrade. Auth-chain identity is Task 3 — for this task, connections get the same treatment UDS-less anonymous loopback logic would give (reuse `authenticate_network`-style gating with peer addr; wire the full HTTP context in Task 3).

**Acceptance:**
- [ ] Daemon binds `--listen ws://127.0.0.1:0`; a Rust `wrpc-websockets` client completes `meta.version` and a full `attach` round-trip (snapshot received, input echoed, detach) against it in integration tests
- [ ] Origin rules covered by tests: same-host pass, allowlisted pass, mismatch rejected, no-Origin pass
- [ ] Non-loopback peer without auth chain is rejected (parity with WT behavior, `serve/auth.rs:30-36`)
- [ ] Graceful shutdown drains WS connections like other listeners

### Task 3: HTTP identity — `TransportContext::Http` + tailscale-serve backend — **sonnet** (blocked by Task 2)

**Files:** create `auth/tailscale_serve.rs`; modify `auth/mod.rs`, `serve/transport/websocket.rs`, `config/{args,mod}.rs` (`--auth tailscale-serve`), `daemon.rs` (chain construction ~`daemon.rs:62-82`); tests alongside existing auth tests.

**Outcome:** WS upgrades run the real `AuthChain` with a new `TransportContext::Http { peer_addr, headers }`. The `tailscale-serve` backend reads `Tailscale-User-Login` (+ display name) and yields that identity **only when `peer_addr` is loopback**; composes first-success-wins with existing backends. Existing `tailscale` (whois) backend keeps working for direct WS connections from tailnet IPs.

**Acceptance:**
- [ ] Tests: header + loopback → identified; header + non-loopback → header ignored (falls through chain); no chain + loopback → anonymous; no chain + non-loopback → rejected
- [ ] `--auth tailscale-serve` selectable alone or alongside `tailscale`
- [ ] Identity lands on the session's attached-client records the same way WT identities do

### Task 4: `--web-ui` serving + `/cairn.json` + embed feature — **sonnet** (blocked by Task 2; parallel-safe with Task 3)

**Files:** modify `serve/http.rs`, `config/{args,mod}.rs`, daemon `Cargo.toml` (feature `web-ui`, off by default; embed via `include_dir` or equivalent); tests in `crates/cairn-daemon/tests/`.

**Outcome (per spec sections "Web UI serving" and "/cairn.json"):** bare `--web-ui` attaches SPA routes to every `ws://` listener (startup error if none); `--web-ui=host:port` binds a dedicated SPA-only HTTP listener instead; `--web-dir` overrides embedded assets and works without the feature; unknown paths fall back to `index.html`; `/cairn.json` (CORS `*`, served on `ws://` listeners regardless of `--web-ui` and on the dedicated listener) reports `endpoints.websocket: "/ws"` and `endpoints.webtransport: { url, certHash }` per configured listeners, certHash present only for the self-signed cert path.

**Acceptance:**
- [ ] Integration tests cover: flag validation error, both serving forms (use `--web-dir` with a fixture dir — no real SPA exists yet), SPA fallback, `/cairn.json` contents for listener combinations (ws-only, ws+wt, wt+dedicated-ui)
- [ ] `cargo build` without the feature requires no npm artifacts; with `--features web-ui` but no `cairn-web/build` output, the build fails with a clear message

### Task 5: cairn-web scaffold + protocol layer — **opus** (no Rust dependencies; parallel-safe with Tasks 1–4)

**Files:** create `cairn-web/` per File Structure (SvelteKit + adapter-static + TS + vitest scaffold folded in — no separate bootstrap task), `vendor/` tarball via `npm pack` in `../wrpc/js` (+ `SOURCE.md` recording the wrpc commit), and the whole `src/lib/protocol/` layer.

**Interfaces produced (later tasks consume these — keep names):** `wit.ts` exports type descriptors mirroring `crates/cairn-protocol/wit/cairn.wit` exactly (that file is the authority); `types.ts` exports TS interfaces (`SessionInfo`, `SessionSpec`, `ExitStatus`, `ServerEvent`, `ClientEvent`, `AttachInit`, `CairnError`, …); `transport.ts` exports `type Dialer = () => Promise<Transport>`; `ws.ts` exports `wsDialer(url: string): Dialer`; `wt.ts` exports `wtDialer(url: string, certHash?: string): Dialer` (one WT session cached, one bidi stream per dial); `client.ts` exports `DaemonClient` with typed methods for all 14 WIT functions — `attach(id, init, clientEvents: AsyncIterable<ClientEvent>): AsyncIterable<ServerEvent[]>`, `wait(id): Promise<ExitStatus>`, `logs(...)`: async iterable, unary methods as promises that reject with `CairnError` on `result::err`.

**Testing note (judgment area):** the SDK ships `accept()` — unit-test `DaemonClient` in-process against an in-memory `Transport` pair with a fake `accept()`-based server, covering value round-trips for every method including streams. No daemon needed here; real interop is Task 6's gate.

**Acceptance:**
- [ ] `npm run check` (svelte-check/tsc) and `npm test` green; biome + svelte-build pre-commit hooks pass
- [ ] Zero framework imports anywhere under `protocol/`
- [ ] In-process round-trip tests for all sessions/meta methods incl. attach event flow in both directions

### Task 6: Wire interop gate — JS SDK ↔ real daemon — **opus** (blocked by Tasks 2 and 5; Task 3 not required)

**Files:** create `cairn-web/tests/integration/` + `vitest.integration.config.ts` + a harness that builds/spawns `cairn-daemon --listen ws://127.0.0.1:<port>` on a temp runtime dir (prior art: the failed branch had a daemon-spawning vitest harness — reuse ideas, not code).

**Outcome:** This is the milestone v1 never reached — proof the official JS SDK interoperates with the daemon over `ws://` before any UI exists. Cover every unary method, `logs` (tail + follow), `wait`, and a full `attach` (snapshot → keystroke input → echoed output → resize → detach) against a real PTY session.

**Acceptance:**
- [ ] All 14 WIT functions exercised end-to-end from Node against the real daemon; error paths too (inspect unknown id → `CairnError` with daemon error code)
- [ ] **CPU regression check (spec "Testing" #4):** with a JS client attached to a quiet session, sampled daemon CPU over ≥5s is ~idle (assert well under 10% of one core; judgment on sampling mechanics)
- [ ] Suite runs via `npm run test:integration` and documents its daemon-binary prerequisite

### Task 7: App shell, stores, session views — **sonnet** (blocked by Task 6)

**Files:** create `src/lib/stores/*`, `src/routes/*` (layout, `/sessions`, `/sessions/new`), `src/lib/components/{SessionList,CreateSession,…}.svelte`; endpoint selection wiring (fetch `/cairn.json` → prefer same-origin WS → WT fallback → manual endpoint entry persisted in localStorage, `?endpoint=` override). The manual entry here is a minimal fallback input only — the full standalone-hosting screen (daemon-URL bootstrap, cert-hash field) ships in Task 9.

**Outcome:** Working SPA (minus terminal): responsive session list (cards narrow / table wide; status, name, command, attached count, relative created time, exit code), create form (primary fields + collapsed advanced section mapping the full `SessionSpec`), connection indicator, reconnect with jittered exponential backoff capped at 10s living in the stores layer (`DaemonClient` stays stateless). Mobile-first per spec "Responsive Design" of the v1 spec's UI-views section (list/create/detail behaviors carried over by reference in the v2 spec).

**Acceptance:**
- [ ] vitest unit tests: reconnect/backoff state machine, endpoint-selection precedence, store refresh on reconnect
- [ ] Manual check against a live daemon (`--web-ui --web-dir` or vite dev proxy — implementer's judgment): list, create, kill, rename round-trips work from the browser
- [ ] `npm run build` emits a static site the daemon can serve (verify Task 4's serving path with the real build once, replacing the fixture-dir-only coverage)

### Task 8: Terminal + attach — **opus** (blocked by Task 7)

**Files:** create `src/lib/components/Terminal.svelte`, `/sessions/[id]` route + detail chrome; wterm deps (`@wterm/dom`, `@wterm/ghostty` 0.3.x).

**Outcome (spec "Terminal & attach flow"):** measure container → attach with correct initial cols/rows → snapshot as first paint → output batches append → keystrokes as `input`, `ResizeObserver`-debounced (~100ms) `resize`, `detach` + queue close on unmount; `exited` overlay with status; in-band `error` (`client.kicked`/`client.lagged`) surfaced with a reattach action; exited-session overlay on the detail view. Backpressure rules: no polling loops, socket writes await drain. Full-viewport terminal with minimal chrome on mobile; wterm stays fully encapsulated in `Terminal.svelte`.

**Acceptance:**
- [ ] Browser-verified against a live daemon: interactive shell session usable (typing, TUI app rendering, resize, detach/reattach, exit overlay) — chrome-devtools MCP or manual, evidence in the task report
- [ ] Attach lifecycle unit-tested where practical (event queue open/close, debounce, overlay state) without requiring a browser
- [ ] Idle attached tab holds daemon CPU at ~idle (spot-check; the automated gate lives in Task 6)

### Task 9: `<cairn-terminal>` web component + standalone hosting — **sonnet** (blocked by Task 8)

**Files:** WC compile entry (Svelte `customElement`) + a separate Vite build target/entry for the element bundle; manual endpoint screen component; docs (`cairn-web/README.md`: daemon-served vs standalone deployment, `--ws-origin` requirement for standalone, WC usage).

**Outcome:** `Terminal.svelte` additionally compiles to `<cairn-terminal>` with the spec's contract (attributes `session-id`/`endpoint`/`font-size`/`font-family`; properties `.client`/`.sessionId`; events `cairn-attached`/`cairn-detached`/`cairn-exited`; shared-client and standalone modes). Standalone static hosting works end-to-end: manual endpoint screen accepts a daemon base URL (fetches its CORS-open `/cairn.json`) or a direct `ws://`/`wss://` URL, WT cert-hash field for the self-signed case.

**Acceptance:**
- [ ] A plain HTML page (no Svelte app) embeds `<cairn-terminal session-id=… endpoint=…>` against a live daemon and gets a working terminal (browser-verified, `--ws-origin` set)
- [ ] Standalone bootstrap-from-daemon-URL flow browser-verified
- [ ] WebTransport path browser-verified once (daemon with `https://` listener, browser client using `wt.ts` via `/cairn.json` or manual entry) — Node has no WebTransport, so this is the only end-to-end WT check in the plan
- [ ] Docs cover both deployment modes and the origin-allowlist requirement
